## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Legal / Criminal Justice (Forensic Science)

**User Population:** Forensic examiners (Peritos forenses) in LATAM criminalistics laboratories generating expert reports for court.

**Stakes Level:** Critical

**Output Consequence:** If the generated report contains hallucinations, incorrect terminology, or weak chain-of-custody linkage, it can lead to wrongful convictions, dismissal of valid evidence, and legal liability for the signing expert. The output directly impacts human liberty and judicial integrity.

### What Domain Experts Evaluate Against

Dimension: Procedural Factual Grounding
Good (expert accepts): Every technical finding (e.g., number of minutiae matched, enhancement parameters) maps 1:1 with the system's database records and audit logs.
Bad (expert flags): The text describes a ridge ending or bifurcation that is not present in the algorithm's actual output, or invents a metric to sound more conclusive.
Stakes: Critical
Source: Forensic Science Standards & Rules of Evidence

Dimension: Tone and Legal Admissibility (Lenguaje Judicial)
Good (expert accepts): The report uses objective, definitive, and legally sanctioned terminology (e.g., "puntos caracterГ­sticos coincidentes", "correspondencia dactiloscГіpica").
Bad (expert flags): The text uses conversational, speculative, or probabilistically ambiguous phrasing (e.g., "parece ser", "es muy probable que", "yo creo").
Stakes: High
Source: Practitioner knowledge (Peritos) and judicial precedents on expert testimony

Dimension: Chain of Custody Integrity
Good (expert accepts): The report accurately cites the digital hash, exact timestamp, and original case/file identifiers without alteration or summarization.
Bad (expert flags): The report summarizes, paraphrases, or truncates the evidence identifiers, breaking the explicit link between physical evidence and the conclusion.
Stakes: Critical
Source: Chain of Custody Regulations / Criminal Procedure Code

### Known Failure Modes in This Domain

- **Synergistic Hallucination:** The LLM invents technical details (e.g., adding non-existent matching minutiae) to make a weak match sound more legally definitive or complete.
- **Tone Drift:** The LLM slips into an "assistant" persona (e.g., "AquГ­ estГЎ el resumen del peritaje...") instead of generating a strict, impersonal legal document.
- **Reference Contamination (Cross-Case Bleeding):** When summarizing, the LLM hallucinates case numbers, dates, or subject names from its training data or previous context instead of strictly using the retrieved SQL data.
- **Over-extrapolation of Confidence:** Stating that a fingerprint definitively belongs to a specific person, whereas forensic standards require stating that the latent print and the reference print originate from the *same source*.

### Regulatory / Compliance Context

- **CГіdigo Procesal Penal:** Requires expert reports to be objective, scientifically reproducible, and personally validated by the sworn expert. The AI acts only as a drafting tool; the expert holds ultimate liability.
- **Reglamento de Cadena de Custodia:** Strict requirements on the non-repudiation, documentation of enhancements, and traceability of digital evidence manipulation.

### Domain Expert Roles for Evaluation

|------|----------------------|
| Perito Forense Senior | Reference dataset labeling (providing gold-standard dictГЎmenes) / Rubric calibration / Evaluating tone and legal admissibility |
| Asesor Legal / Fiscal | Edge case review for court admissibility / Production sampling to ensure no procedural violations |

## 3. Framework Quick Reference

> Fetched from official docs by `gsd-ai-researcher`. Distilled for this specific use case.

### Installation

### Core Imports
```python
from llama_index.core import SQLDatabase
from llama_index.core.query_engine import NLSQLTableQueryEngine
from llama_index.llms.ollama import Ollama
from sqlalchemy import create_engine
from pydantic import BaseModel, Field
```

### Entry Point Pattern
```python
# 1. Initialize local LLM via Ollama
llm = Ollama(
    model="llama3.1:latest",  # or custom fine-tuned model
    request_timeout=120.0,
    context_window=8000,      # Crucial for local models to prevent OOM
)

# 2. Connect to Database (Must use READ-ONLY credentials)
engine = create_engine("postgresql+psycopg2://user:pass@localhost:5432/db")
sql_database = SQLDatabase(engine, include_tables=["peritajes", "evidencia"])

# 3. Create the Text-to-SQL Query Engine
query_engine = NLSQLTableQueryEngine(
    sql_database=sql_database,
    tables=["peritajes", "evidencia"], # Explicitly specify to save context window
    llm=llm,
    synthesize_response=True
)

# 4. Query
response = query_engine.query("Genera un resumen del peritaje nГєmero 450 incluyendo la evidencia asociada.")
print(response)
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `NLSQLTableQueryEngine` | Engine that translates Natural Language to SQL, executes it, and synthesizes a response. | When allowing users to query structured relational data (PostgreSQL) using natural language. |
| `SQLDatabase` | LlamaIndex wrapper around a SQLAlchemy engine. | To expose your database schema and tables to the query engine. |
| `Ollama` | Local LLM integration class. | When connecting to locally hosted models (e.g., Llama 3) instead of API-based providers. |
| `as_structured_llm` | Wrapper to enforce JSON/schema adherence. | When you need the generation pipeline to output a specific Pydantic object (e.g., a formal report structure). |

### Common Pitfalls
1. **Security Risk (SQL Injection):** Executing arbitrary SQL generated by an LLM is dangerous. **Always** use read-only database credentials and restrict access roles when using `NLSQLTableQueryEngine`.
2. **Context Window Overflow:** If you do not explicitly pass the `tables=["..."]` parameter to `NLSQLTableQueryEngine`, it will attempt to inject your *entire* database schema into the prompt, failing spectacularly for large databases.
3. **Local LLM OOM:** When using Ollama/vLLM, always explicitly set the `context_window` parameter (e.g., 8000). If omitted, defaults may cause the local model to crash on long schema contexts.
4. **Non-Deterministic SQL:** The LLM might hallucinate column names or relationships. Use `text_to_sql_prompt` to provide specific schema hints and valid joins to the LLM.

## 4. Implementation Guidance

**Model Configuration:**
- **Provider**: Ollama (Local)
- **Model**: `llama3.1:latest` (or fine-tuned equivalent for Spanish legal/forensic text)
- **Parameters**: `temperature=0.1` (strict adherence for legal text), `context_window=8000`, `request_timeout=120.0`.

**Core Pattern:**
Hybrid Retrieval and Generation. 
1. Use `NLSQLTableQueryEngine` to extract specific factual data from PostgreSQL (e.g., case details, evidence list).
2. Pass the extracted factual data as context to a Content Generation prompt.
3. Use `llm.as_structured_llm()` to ensure the generated dictamen pericial adheres to the strict legal Pydantic schema.

**Tool Use:**
- **Text-to-SQL**: `NLSQLTableQueryEngine` is essentially a tool that executes SQL. Ensure SQLAlchemy is configured to use a scoped, read-only session.

**State Management:**
- Stateless execution. Each report generation is a single pipeline execution (SQL Extraction -> Prompt -> Structured Generation). State is maintained in the application database, not the LLM framework.

**Context Window Strategy:**
- Limit SQL schema context by strictly providing `tables=["..."]` in the query engine.
- If the SQL result set is too large (e.g., 1000s of rows of evidence logs), implement a row limit in the SQL prompt or summarize the evidence before generating the final report.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs вЂ” independent of framework choice.

### Structured Outputs with Pydantic

To enforce that the generated "dictamen pericial" contains all legally required sections, define a Pydantic model and use LlamaIndex's `as_structured_llm`.

```python
from pydantic import BaseModel, Field
from typing import List

class Evidencia(BaseModel):
    id: str = Field(..., description="ID de la evidencia")
    descripcion: str = Field(..., description="DescripciГіn detallada del hallazgo")

class DictamenPericial(BaseModel):
    """Estructura formal del dictamen pericial."""
    numero_caso: str = Field(..., description="NГєmero de caso o expediente")
    resumen_hechos: str = Field(..., description="Resumen objetivo de los hechos investigados")
    hallazgos: List[Evidencia] = Field(..., description="Lista de evidencias y hallazgos tГ©cnicos")
    conclusion: str = Field(..., description="ConclusiГіn tГ©cnica del perito")
    nivel_confianza: float = Field(..., description="Nivel de confianza de la conclusiГіn (0-1)")

# Usage with LlamaIndex
sllm = llm.as_structured_llm(DictamenPericial)
# Use a custom prompt template that forces the LLM to output JSON matching this schema
prompt = PromptTemplate("Genera el dictamen pericial para el caso {caso_id} basado en estos datos: {datos}")
report = sllm.complete(prompt.format(caso_id="450", datos=sql_results)).raw
```
*Retry Logic*: If JSON validation fails, LlamaIndex handles schema enforcement internally (especially via Ollama's native JSON mode). If it still fails, implement a manual retry loop (max 3 retries) catching `pydantic.ValidationError`, passing the error message back to the LLM.

### Async-First Design

Use the asynchronous equivalents in LlamaIndex to prevent blocking the FastAPI event loop during long local LLM generation. 
- Use `aquery` instead of `query` on the `NLSQLTableQueryEngine`.
- Use `acomplete` instead of `complete` on the LLM.
- **Common Mistake**: Wrapping synchronous LlamaIndex calls in standard synchronous FastAPI routes will lock up the server for minutes while the local LLM generates the report.

### Prompt Engineering Discipline

- **Separation**: Separate the System Prompt (defining the role: "Eres un perito informГЎtico experto en la legislaciГіn de Nicaragua...") from the User Prompt (the actual data to format).
- **Token Limits**: Always set `max_tokens` (e.g., 2000) for the generation step to prevent runaway local generation loops.
- **Language**: Explicitly instruct the LLM to generate responses in formal, legal Spanish.

### Context Window Management

For Text-to-SQL, context window overflow is the biggest risk. 
- **Compaction**: Do not pass raw database rows directly if they exceed 50. Use SQL aggregations (`GROUP BY`, `COUNT`) within the Text-to-SQL instructions.
- **Truncation**: If evidence logs are massive, implement an intermediate map-reduce summarization step over the evidence before feeding it into the final Dictamen generation prompt.

### Cost and Latency Budget

- **Cost**: Since using local Ollama/vLLM models, API cost is $0, but compute cost (GPU time) is high.
- **Latency**: Local models may take 10-30 seconds to generate a full report.
- **Caching**: Implement exact-match caching (e.g., using Redis) for identical Text-to-SQL queries. Do not re-run the LLM SQL generation if the same question is asked about the same case.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| **Output Structure Validity** | PASS: JSON output matches `DictamenPericial` Pydantic schema perfectly.<br>FAIL: Missing required fields, wrong data types, or invalid JSON. | Code (Pydantic validation) | Critical |
| **Procedural Factual Grounding** | PASS: Every technical finding, case ID, and metric maps exactly to the provided SQL context.<br>FAIL: Hallucinates evidence, invents minutiae, or fabricates case details (Synergistic Hallucination). | LLM Judge (RAGAS Faithfulness) + Code (ID extraction match) | Critical |
| **Chain of Custody Integrity** | PASS: Hash values, timestamps, and case identifiers are included verbatim.<br>FAIL: Hash values are shortened, timestamps generalized, or identifiers paraphrased. | Code (Exact string/regex matching of hashes and IDs) | Critical |
| **Tone & Legal Admissibility** | PASS: Uses objective, definitive, legally sanctioned terminology (e.g., "puntos caracterГ­sticos coincidentes").<br>FAIL: Uses conversational, speculative, or assistant-like phrasing (e.g., "parece ser", "aquГ­ estГЎ el resumen"). | LLM Judge + Code (Regex keyword deny-lists) | High |
| **Epistemic Certainty Limits** | PASS: States prints originate from the *same source* based on minutiae.<br>FAIL: Definitively asserts identity without forensic caveats (Over-extrapolation of confidence). | LLM Judge | High |

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Schema Compliance** | `pydantic.ValidationError` during generation | Block & Retry (max 3 times), then Escalate to user ("Error de Formato"). |
| **Chain of Custody Matcher** | Missing exact DB Hash or Evidence ID in generated text | Block (Return "Error: PГ©rdida de Cadena de Custodia en el reporte"). |
| **Tone Denylist** | Regex match for conversational terms ("yo creo", "aquГ­ tienes", "parece") | Flag (Highlight warnings for the Perito before export/signing). |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Synergistic Hallucination** | Random 10% of generated reports evaluated via RAGAS | Review prompt constraints; add few-shot examples; fine-tune local model. |
| **Tone Drift** | Sample cases where Perito heavily edited the generated text | Calibrate local model temperature and system prompt instructions. |
| **Context Overflow** | SQL queries returning > 100 rows or exceeding token limits | Implement SQL row limits or intermediate map-reduce summarization. |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix (Self-hosted local Docker container to maintain air-gapped/on-premise constraints).

**Key Metrics to Track:**
1. **Generation Latency:** Time to generate the report (Seconds). Since local GPUs vary, this is critical.
2. **Context Window Saturation:** Percentage of max tokens used per request.
3. **Acceptance vs. Edit Rate:** Edit distance between the LLM-generated draft and the final signed report saved by the Perito.
4. **Schema Retry Rate:** Frequency of Pydantic validation failures requiring automatic LLM retries.

**Alert Thresholds:**
- **Latency > 45s**: Alerts admin that the local GPU is struggling or queue is saturated.
- **Edit Rate > 30%**: Alerts team that generated quality is poor and requires Perito intervention.
- **Context Saturation > 90%**: Alerts team of risk of truncated context/OOM errors.

**Smart Sampling Strategy:**
- **High Edit Distance**: Automatically route 100% of reports where the Perito changed > 20% of the text to the offline flywheel for prompt refinement.
- **Retry Spikes**: Route all reports that required 2+ schema retries.
- **Random Audit**: Route 5% of randomly selected "accepted as-is" reports to Legal/Audit for quality assurance against procedural violations.
